Flow 1 — user search & track on-demand · HTTPS
Browser hits the API; the API reads from GCS for search results and Firestore for tracked flights.
Flow 2 — live ingest every 15m
Cloud Scheduler fires the scheduler service, which fetches fresh Amadeus fares and writes them to GCS + Firestore.
Flow 3 — drop detection & email alert daily · 08:00
Scheduler reads each tracked flight, pulls current price from GCS, compares to baseline, and emails on drop (dedup 24h in Firestore notification_log).

Cross-cutting — Secrets

Both Cloud Run services mount AMADEUS_CLIENT_ID/SECRET, SENDGRID_API_KEY, and SCHEDULER_TOKEN from GCP Secret Manager at startup.

Cross-cutting — Deploy

Single Dockerfile → Cloud Build → Artifact Registry. Two Cloud Run services share the image; SERVICE_MODE env var selects api vs scheduler entrypoint.

Cross-cutting — Seeding

scripts/model_pipeline (offline) pulls Amadeus + BTS DB1B, builds baselines, and writes flight_data.csv to GCS. scripts/flight_fetch* backs this up with SerpAPI.
